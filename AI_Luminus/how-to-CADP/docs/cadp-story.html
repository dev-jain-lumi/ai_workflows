What CADP Means

A Consumer-Aligned Data Product (CADP) is the consumer-facing contract in the data product landscape. It does not exist to mirror source systems; it exists to deliver stable, understandable, analytics-ready outputs for business consumption.

Teams commonly discuss three product families in this context: source-aligned products (SADP), aggregate products (ADP), and consumer-aligned products (CADP). The CADP layer should hide upstream complexity and present reliable semantics.

Tool and Technology Landscape

A CADP journey is enabled by a coordinated stack where each technology has a clear role. The point is not to use tools independently, but to run them as one controlled delivery chain.

Lifecycle in Plain Language

  1. Define and govern purpose so ownership and boundaries are clear.
  2. Provision baseline technical workspace and access.
  3. Confirm setup works before deeper coding efforts begin.
  4. Secure required upstream SADP agreements.
  5. Model and test CADP logic iteratively in development.
  6. Commit and deploy through controlled CI/CD plus orchestration.
  7. Register and publish for discoverable, governed consumption.

This is a delivery sequence, not a strict one-pass pipeline. Teams should expect loops in development and validation, while keeping publication gates controlled.

Modelling Expectations

CADP modelling is strongest when it starts from consumer questions and target grain, then maps backward to available source-aligned inputs. Typical implementation separates staging, intermediate, dimensions, and facts. Tests are part of the product contract, not optional decoration.

The current examples in this repository intentionally encourage explicit grain definitions, key quality checks, and measure reconciliation before publication.

1. Context

The session intent was to position tools, align on the CADP concept, and confirm setup readiness before deep implementation sessions. This is an important operating pattern on NEO: first establish access, project readiness, and shared language, then move into detailed modelling and coding.

Practically, this means teams should not wait for perfect conditions before learning, but they should avoid jumping into production-grade coding before environment and ownership foundations are in place.

2. What CADP Is

A Consumer-Aligned Data Product (CADP) is a governed, consumer-facing data product. It translates source-oriented data into stable business entities and measures that are easier to consume in analytics, reporting, and downstream decision processes.

  • It is consumer-first, not source-first.
  • It should expose a stable data contract over time.
  • It is expected to be testable, explainable, and discoverable.

3. Data Product Taxonomy

SADP Source-Aligned Data Product. Raw or near-raw source data ingestion and exposure.
ADP Aggregate Data Product. Integration and aggregation layer across source-aligned products.
CADP Consumer-Aligned Data Product. Business-oriented, consumption-ready model and contract.

Typical progression is from source fidelity to consumer usefulness, while preserving governance and ownership.

6. Modelling Approach

Layered model strategy

  • Staging: normalize and clean source-aligned inputs.
  • Intermediate: centralize reusable business logic.
  • Dimensions: stable business entities and descriptive attributes.
  • Facts: measurable events/snapshots with explicit grain.

Design emphasis

  • Document grain before implementation.
  • Prefer explicit contracts over implicit assumptions.
  • Ensure measures reconcile with trusted references.
  • Avoid downstream consumers rebuilding source complexity.

7. Quality and Validation

CADP quality is both technical and business-facing. Technical validity without business clarity is not enough, and business language without technical controls is fragile.

Technical checksDBT runs/tests, null checks, duplicate grain checks, relationship checks.
Business checksGrain clarity, measure consistency, understandable semantic naming.
Governance checksAccess agreements, metadata registration, ownership and support route.

8. Publish and Operate

Publishing is the transition from development output to governed consumer product. Local runs validate logic, but production trust comes from controlled deployment, registration, and discoverability in the portal.

  • Monitor pipeline reliability and run outcomes.
  • Manage changes through versioned workflows.
  • Coordinate breaking changes with consumers early.
  • Keep metadata and support ownership up to date.
