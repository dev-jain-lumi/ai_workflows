Chat is not a code generator

Three uses that change how you work:

Explain → Challenge → Ask for the missing test

The slash command vocabulary

Command Use it for
/explain Understanding what code does
/fix Repairing a specific failure
/doc Generating NumPy docstrings
/tests Generating a test suite

Warning

/explain describes intent, not correctness. It will miss logical bugs.

“These aren’t magic. They’re structured prompts with context injection built in. Understanding that they’re just prompts helps you use them better — and know when to add more detail.”

The callout is the key lesson of this module. Emphasise it: “/explain will tell you what the code is trying to do. It will not tell you if it’s doing it wrong.” That’s what the exercise will prove.

/fix — precision matters

Weak

/fix

Chat guesses what to fix. Usually wrong.

Strong

/fix summer records in compute_cet_timestamps
must convert Europe/Brussels local time to
fixed CET (UTC+1). Right now cetStartDate
and cetStarttime copy startDate/startTime
through unchanged, which breaks summer rows
and midnight rollback.

Chat fixes the right thing.

“The single most common mistake with /fix is not describing the failure. Chat has no idea what ‘wrong’ means without a description.”

Demo this live if time allows: select localise_to_cet, type /fix with no description, show the result. Then add the description. Show how different the output is.

The bug in cet_converter.py

def compute_cet_timestamps(records):
  result = []
  for r in records:
    r = dict(r)
    r["cetStartDate"] = r["startDate"]   # BUG: ignores CEST rollback
    r["cetStarttime"] = r["startTime"]   # BUG: summer should subtract 1h
    result.append(r)
  return result

Winter data passes because local time already equals fixed CET.

Summer rows should subtract 1 hour, and midnight summer rows roll back to the previous CET date.

Fall-back order B is a separate rule: the repeated 02:00 hour is already CET.

Do NOT reveal this until after participants have attempted /explain on compute_cet_timestamps. This slide is for the debrief, not the introduction.

The key teaching point: “/explain described the function correctly and missed the bug entirely.”

Ask: “How many of you have had a bug like this in production — correct-looking output, wrong calculation underneath?” Usually several hands. This validates why the exercise matters.

Exercise 2 — Workflow sprint (40 min)

Work through M2_CET_BUG.ipynb and curve_pipeline/cet_converter.py in order:

  1. /explain on compute_cet_timestamps — does it describe intent or detect the bug?
  2. Compare the result with scenarios/1-bug-hunt/airflow_logs_fixture.txt
  3. /fix with precise regime context: summer subtracts 1 hour, fall-back order B does not
  4. Re-run the summer and fall-back checks. What improved? What still fails?

Note

/explain is descriptive. Regression output is proof.

Debrief question after: “What would actually have caught this bug earlier?” Answer: a summer checkpoint or a unit test that asserts cetStarttime = startTime - 1h for a summer record and the no-conversion rule for fall-back order B.

  1. Negative quotes are valid input.
  2. Reuse helpers from cet_converter.py instead of reimplementing DST logic.

Plan mode makes TDD practical

Most teams ask Copilot for code too early. The result: tests that prove the implementation you got, not the contract you wanted.

Use plan mode to enumerate the CET regimes before you generate any code:

  1. Winter: no conversion
  2. Summer: subtract 1 hour
  3. Summer midnight: roll back to the previous CET date
  4. Fall-back order B: no conversion

Do NOT implement anything yet."

“Then keep tests/test_validator.py open as a tab while implementing. That turns the tests into live context for Copilot.”

The actual traceback carries more information than any description.

“A traceback contains: the test name (what was being tested), the assertion that failed (what was expected vs actual), and the exact line. A description like ‘it’s broken’ contains none of that.”

“Paste the full traceback. Select the function. Then /fix. That’s the loop.”

The negative prices litmus test

If your generated CurveValidator flags negative quotes, your tests should catch it.

def test_negative_quote_is_valid():
    record = {
    "quotedDate": "2026-02-05",
    "startDate": "2026-02-05",
    "startTime": "13:00:00",
    "order": "A",
    "quote": "-45.2",
    "unit": "EUR/MWH",
    "type": "INPUT",
    "cetStartDate": "2026-02-05",
    "cetStarttime": "13:00:00",
    }
    result = validator.validate_record(record)
    assert result.is_valid
    assert result.errors == []

If Copilot’s implementation fails this test — that’s the lesson working.

“This test is the red thread of the whole day. It first appeared in the instructions you wrote in M3. Now it’s a test. In M5 the agent may fail it on first try. In M6 the DAG’s SODA and validator path must still respect it.”

“If Copilot generates an implementation that filters negative prices and THIS test catches it — that’s exactly the lesson. Your tests protected you from domain ignorance in the model.”

Agent + Parallel Worktrees + SODA

Note

Facilitator notebook: facilitator/M5_facilitator.ipynb

Transition: “Everything so far has been single-turn or single-file. Now we split the work into two parallel change streams and let agents operate inside explicit boundaries.”

Key teaching moment: worktrees are not just convenience. They are the mechanism that keeps independent agent changes reviewable and mergeable.

What agent mode actually does

Agent mode plans and executes sequences of edits across multiple files + terminal.

It makes confident mistakes.

The review obligation: every agent-produced line must be read before commit. Not because agents are bad — because they have no domain knowledge.

Important

The test suite from Module 4 and the worktree split are the safety nets that catch it.

“Agent mode is powerful. It’s also the place where people get into trouble by trusting the output too much. A 3-file change that looks coherent can have a subtle domain error in it — like filtering negative prices — that won’t crash but will silently corrupt results.”

“The test suite is not optional. It’s the contract the agent has to satisfy.”

Show the Copilot Edits panel briefly — point out the undo button. “This is your safety net at the UI level. The tests are your safety net at the logic level.”

Two worktrees in parallel

Scenario 1 — bug hunt Worktree: .worktrees/scenario-1

Context: - scenarios/1-bug-hunt/airflow_logs_fixture.txt - curve_pipeline/cet_converter.py - summer and fall-back regression behaviour

Goal: fix the CET bug without breaking order B.

Scenario 2 — new validation feature Worktree: .worktrees/scenario-2

Context: - scenarios/2-new-feature/spec.md - curve_pipeline/validator.py - tests-first workflow from Module 4

Goal: implement CurveValidator and keep duplicates scoped to quotedDate.

The contrast is the lesson: same agent skill, very different review burden.

“Start Session 1 first. While it’s running — agent mode can take 2-3 minutes — start Session 2 in a new tab. You’ll review both when they finish.”

“The contrast is intentional. Session 1 has rich context and tests to validate against. Session 2 has a spec and nothing else. You’ll feel the difference in how much manual review Session 2 requires.”

Agent prompt structure

Goal: implement Scenario 2 in `.worktrees/scenario-2`

Constraints:
  - reuse `_is_fall_back_day` and `validate_cet_record`
  - negative quotes are valid
  - do not modify `curve_loader.py`

Output:
  - `curve_pipeline/validator.py`
  - `tests/test_validator.py`
  - green `pytest tests/test_validator.py -v`

Acceptance:
  - `order=B` only on fall-back day
  - duplicates are scoped by `quotedDate`

Finish with /review-cet-function before accepting the final diff.

“Notice the prompt structure: Goal, Constraints, Output, Acceptance. The constraints section is where your domain knowledge lives — the agent has no idea that negative quotes are valid or that it should reuse helpers unless you tell it.”

If participants already have a helper prompt in their notebook flow, that’s fine. The teaching point is still the same structure.

Session 1 review checklist

After the agent finishes — before accepting anything:

Every participant should find at least 2 things the agent assumed wrongly.

“Go through this checklist out loud with the room after everyone’s agent has finished. Ask: who found something wrong with item 3? Item 4?”

Common agent failures: - Using pytz instead of standard library - Treating negative quotes as warnings - Reimplementing fall-back logic instead of reusing helpers - Duplicate detection ignoring quotedDate - Fixing summer and accidentally shifting order B

“If your tests all pass — great. But still go through the checklist. Tests are necessary but not sufficient.”

SODA checks are a different safety net

Three things /add-soda-checks should give you:

  1. missing_count = 0 for cetStartDate and cetStarttime
  2. valid_values checks for type and order
  3. A row-count discussion for spring_forward

Live follow-up:

soda scan -d curves -c soda_checks/curve_quality.yml data/curves_spring_forward.csv

The discussion point from the facilitator guide matters here: 23 rows on spring-forward day is correct. If SODA flags it, that is a threshold problem, not a CET code bug.

Debrief: “What does SODA catch that regression does not? What can regression catch that SODA cannot?”
